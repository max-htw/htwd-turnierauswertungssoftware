== Einleitung

Dieses Testkonzept beschreibt die Vorgehensweise zur Planung, Durchführung und Dokumentation der Tests für die Volleyballturnier-Software. Ziel ist es, die Qualität der Software sicherzustellen und zu gewährleisten, dass alle Anforderungen erfüllt werden.

== Testziele

Sicherstellung der korrekten Erfassung und Auswertung von Spielergebnissen

Überprüfung der Benutzerfreundlichkeit auf mobilen Geräten
// War das eine Anforderung?
Prüfung der Stabilität im Offline-Betrieb 

Sicherstellung der korrekten Generierung von Spielplänen und Ergebnislisten (z. B. als PDF)

== Testgegenstand

Der Testgegenstand ist die entwickelte Webanwendung mit folgenden Technologien:

Backend: Java und SQLite

Frontend: HTML und CSS

Weitere Tools: PDF-Export

== Testarten

[cols="2,5"] 
|=== 
| Testart | Beschreibung

| Funktionaler Test | Prüfung, ob die Software alle funktionalen Anforderungen erfüllt (z. B. Ergebnisseingabe, Anzeige der Tabellen) | Usability-Test | Bewertung der Benutzerfreundlichkeit, insbesondere auf Smartphones | Komponententest | Einzeltests der Java-Methoden (Unit Tests) | Integrationstest | Zusammenspiel von Frontend und Backend | Systemtest | Test des Gesamtsystems unter realen Bedingungen 
|===

== Testvorgehen

. Erstellung von Testfällen auf Basis der Anforderungen und User Storys . Manuelle Durchführung der Tests (ggf. teilautomatisiert mit JUnit im Backend) . Dokumentation der Ergebnisse in einem Testprotokoll . Rückmeldung an die/den Entwickler bei Fehlern . Wiederholung der Tests nach Fehlerbehebung

== Testumgebung

Testgeräte: Smartphone (Android/iOS), Laptop

Browser: Chrome, Safari

Netzwerkbedingungen: Offline-Tests (kein WLAN), normale WLAN-Verbindung

Testdaten: Beispiel-Spielpläne, fiktive Teams, Beispiel-Ergebnisse

== Testdokumentation

Alle durchgeführten Tests werden in einer Testtabelle dokumentiert

Fehlermeldungen werden ausführlich dokumentiert ( beispielsweise Screenshot) und beschrieben

Protokolle werden im Git-Repository abgelegt

== Abnahmekriterien

Alle kritischen und wichtigen Testfälle müssen erfolgreich bestanden sein

Keine Blocker oder Major Bugs mehr

Software ist stabil im Testbetrieb einsetzbar
